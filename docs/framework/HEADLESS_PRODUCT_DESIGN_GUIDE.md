# ヘッドレスプロダクト設計フレームワーク

> 本ドキュメントは、ヘッドレス型プロダクトを**どう作るべきか**の汎用フレームワーク（抽象）を定義する。
> Java的に言えば、ヘッドレスプロダクトが満たすべき `interface` と `abstract class` の集合。
> 具象プロダクト（IDM等）は本フレームワークを `implements` して構築する。
>
> 関連ドキュメント：
> - [HEADLESS_ARCHITECTURE_AND_GOVERNANCE.md](./HEADLESS_ARCHITECTURE_AND_GOVERNANCE.md) — UIを被せた時のガバナンスフレームワーク
> - [HEADLESS_IDM_APPLICATION.md](./HEADLESS_IDM_APPLICATION.md) — 本フレームワークの具象適用例

---

## 1. 問題ドメインの認識（Problem Domain Recognition）

### 1.1 なぜヘッドレスが必要か

従来のUI一体型プロダクトは以下の構造的問題を抱える：

```
UI一体型の構造的問題：
  ├── UI変更 → ビジネスロジック変更を誘発（密結合）
  ├── テスト → UIテストは脆く、維持コストが高い
  ├── 多チャネル対応 → UIごとにロジック複製が発生
  ├── AI/自動化対応 → UIパース前提の統合は壊れやすい
  └── 保守 → UI起因のリグレッションが業務ロジックに波及
```

### 1.2 ヘッドレス化の判断基準

全てのプロダクトがヘッドレスであるべきとは限らない。
以下のうち **2つ以上** に該当する場合、ヘッドレス化の投資対効果がある：

| 条件 | 判断ポイント |
|------|-------------|
| **多チャネル要求** | LLM / Web UI / CLI / 外部システム など、複数の消費者が存在する（将来含む） |
| **テスタビリティ要求** | 業務ロジックのリグレッション検出が重要（金融・医療・IDM等） |
| **AI統合要求** | LLMによる自律操作・自動化が想定される |
| **長寿命** | プロダクトのライフサイクルが3年以上で、UIの世代交代が起きる |
| **外部提供** | APIを第三者に提供し、顧客が独自UIを構築する余地がある |

### 1.3 ヘッドレス化が不要なケース

```
ヘッドレス化しなくてよい場合：
  ├── ユーザーが単一で、UIが唯一のインターフェース
  ├── プロダクト寿命が短い（PoC、キャンペーン等）
  ├── ロジックが極めて単純で、UI変更がロジックに波及しない
  └── 初期コストを許容できない段階（→ 3.のライフサイクルで段階的に移行）
```

### 1.4 既存UIサービスからのヘッドレス化（機能要件抽出）

既存のUI一体型サービスをヘッドレス化する場合、
UI層に埋め込まれた暗黙の機能要件を抽出する必要がある。

#### 1.4.1 抽出プロセス

```
Step 1: 画面棚卸し
  └── 全画面の一覧を作成し、各画面の「やっていること」をリスト化

Step 2: 操作の分類
  └── 各画面の操作を以下に分類：
      ├── 参照（Read）：データの表示・検索・フィルタ
      ├── 変更（Write）：データの作成・更新・削除
      ├── ワークフロー（Flow）：複数ステップの業務フロー
      └── 表示ロジック（View Logic）：UIでのみ意味がある処理

Step 3: コア候補の特定
  └── 参照・変更・ワークフロー → コア層のAPI候補
      表示ロジック → オペレータ層に残留

Step 4: 暗黙ルールの可視化
  └── UIのバリデーション・条件分岐・権限チェック等
      → コア層のビジネスルールとして明示的に定義

Step 5: API境界の設計
  └── 抽出した操作群をリソース単位でAPIとして設計
```

#### 1.4.2 抽出の判定基準

```
コア層に移すべきもの：
  ├── 「UIが変わっても同じであるべき」処理 → コア
  ├── 「別のUIでも同じルールを適用すべき」バリデーション → コア
  └── 「監査対象になる」操作 → コア

オペレータ層に残すもの：
  ├── 「このUIでだけ意味がある」表示変換 → オペレータ
  ├── 「見せ方の問題」（ページネーション表示方法等）→ オペレータ
  └── 「UX上の利便性」（フォーム自動保存等）→ オペレータ
```

## 2. 層分離契約（Layer Separation Contract）

ヘッドレスプロダクトは、以下の2層に分離しなければならない。

```
┌─────────────────────────────────────────┐
│        オペレータ層（Operator Layer）      │
│  消費者がコアを操作するためのインターフェース │
│  LLM / UI / CLI / 外部システム           │
├─────────────────────────────────────────┤
│        コア層（Core Layer）               │
│  ビジネスロジック・データ・外部連携         │
│  API として公開                          │
└─────────────────────────────────────────┘
```

### 2.1 コア層の契約

コア層は以下を満たさなければならない：

| 契約 | 定義 |
|------|------|
| **決定論性** | 同じ入力に対して必ず同じ出力を返す。推論・確率的処理を含まない |
| **自立性** | オペレータ層が存在しなくても単体で動作する |
| **テスタビリティ** | 全操作がAPIテストで検証可能である |
| **監査可能性** | 全操作の履歴が記録され、追跡可能である |
| **冪等性** | 同じリクエストを複数回送信しても副作用が1回分である |

### 2.2 オペレータ層の契約

オペレータ層は以下を満たさなければならない：

| 契約 | 定義 |
|------|------|
| **翻訳者であること** | ビジネスロジックの判断を行わない。コアのAPIを呼び出すのみ |
| **交換可能性** | 別のオペレータ（LLM→UI、UI→CLI等）に置換してもコアに影響しない |
| **非保持** | 業務データを自身に保持しない。全てコア層経由で取得・保存する |
| **障害独立性** | オペレータ層が停止してもコア層は影響を受けない |

### 2.3 層間の通信契約

```
オペレータ層 → コア層：API呼び出しのみ（DB直接操作禁止）
コア層 → オペレータ層：API レスポンスのみ（コア層からオペレータ層への直接Push禁止）
```

**コア層はオペレータ層の存在を知らない。** コア層から見れば、誰がAPIを呼んでいるかは関係ない。

> **注記：イベント駆動パターンについて**
> 「Push禁止」はコア層がオペレータ層を直接呼び出すことの禁止であり、
> イベント駆動アーキテクチャを否定するものではない。
> コア層がイベントバス / メッセージキューに発行し、
> オペレータ層がそれを購読する構造は、層間の通信契約に違反しない。
> コア層は「誰が購読しているか」を知らないからである。

### 2.4 責務分割ヒューリスティクス（Responsibility Partitioning）

層分離を設計する際、「この処理はどちらの層に置くべきか」の判断基準：

#### 2.4.1 判断フローチャート

```
ある処理を評価する
  │
  ├── UIが変わっても処理ロジックは同じか？
  │     ├── Yes → コア層
  │     └── No  → オペレータ層
  │
  ├── この処理の結果に監査が必要か？
  │     ├── Yes → コア層
  │     └── No  → 次の質問へ
  │
  ├── 別のオペレータ（CLI/LLM/API直接）でも同じ結果を期待するか？
  │     ├── Yes → コア層
  │     └── No  → オペレータ層
  │
  └── この処理はデータの「見せ方」か「作り方」か？
        ├── 見せ方 → オペレータ層
        └── 作り方 → コア層
```

#### 2.4.2 典型的な境界ケース

| 処理 | 判断 | 理由 |
|------|------|------|
| バリデーション（必須チェック等） | コア層 | 全オペレータで同じルール |
| UXバリデーション（リアルタイム入力確認） | オペレータ層 | UX固有。コアでも最終チェックする |
| ページネーション（データ取得） | コア層 | カーソル/オフセットをAPIで提供 |
| ページネーション（表示方法） | オペレータ層 | 無限スクロール/ボタン等はUI判断 |
| 権限チェック | コア層 | API自体が拒否する（オペレータの善意に依存しない） |
| 権限による表示制御 | オペレータ層 | コアが返す権限フラグに基づき表示を変える |
| ソート・フィルタ（ロジック） | コア層 | APIパラメータとして提供 |
| ソート・フィルタ（UI操作） | オペレータ層 | ドラッグ&ドロップ等はUI判断 |
| 通知送信 | コア層 | イベント発火はコア。配信チャネル選択はコアまたは設定依存 |
| 通知表示 | オペレータ層 | トースト/バッジ等はUI判断 |

#### 2.4.3 グレーゾーンの解決原則

```
迷ったらコア層に寄せる。

理由：
  ├── コア層に置けば全オペレータで一貫性が保たれる
  ├── オペレータ層に置くと、オペレータ増加時に重複実装が発生する
  └── 後からコア → オペレータへの移動は容易だが、逆は困難
      （コア層はオペレータを知らないため、依存方向が一方向）
```

## 3. 収束定理（Convergence Theorem）

LLM時代のプロダクト構築アプローチは全てヘッドレスAPIに収束する。

### 3.1 3つのアプローチ

```
アプローチA: LLMが毎回コード生成して実行
  特性：非決定的、テスト不能、コスト高
  結論：商用不可

アプローチB: LLMが事前定義関数（Skills）を選択・実行
  特性：関数は決定的、テスト可能、コスト低
  結論：動作するが密結合

アプローチC: LLMがAPIエンドポイントを呼び出す
  特性：APIは決定的、テスト可能、コスト低、疎結合
  結論：商用品質
```

### 3.2 収束の証明

```
アプローチA → 信頼性のためにSkill化 → アプローチB
アプローチB → 疎結合のためにAPI化 → アプローチC
アプローチC → ここが終着点

∴ A → B → C（全てCに収束する）
```

**アプローチBのSkill関数群 ≒ アプローチCのAPIエンドポイント。**
名前が違うだけで本質は同じ構造。したがって B → C の移行コストは低い。

### 3.3 疎結合（アプローチC）の優位性

| 特性 | 密結合（B） | 疎結合（C） |
|------|-----------|-----------|
| コア障害独立性 | ❌ LLM停止 = サービス停止 | ✅ LLM停止でもAPI継続 |
| オペレータ交換 | ❌ Skill定義の書き直し | ✅ API仕様は不変 |
| 段階的提供 | ❌ LLM必須 | ✅ LLMなし版も可 |
| テスト | △ 統合テスト主体 | ✅ API単体テスト可 |
| ベンダーロックイン | ⚠️ LLM固有のSkill形式 | ✅ OpenAPI標準 |

### 3.4 既存プロダクトの収束パス

既にUI一体型で構築されたプロダクトも、以下の段階でCに収束する。

```
既存UI一体型プロダクト
  │
  ├── Step 1: 機能要件の抽出（1.4参照）
  │     └── UIに埋め込まれたロジックを識別・分類
  │
  ├── Step 2: 内部APIの形成
  │     └── UIから呼ぶ内部関数をAPI形式に整理（まだ同一プロセス内）
  │
  ├── Step 3: API境界の確立
  │     └── 内部APIに仕様書（OpenAPI等）を付与し、正式なコントラクトにする
  │
  ├── Step 4: プロセス分離
  │     └── コア層を別プロセス/別コンテナに切り出し
  │
  └── Step 5: オペレータの多元化
        └── 新規オペレータ（LLM等）を追加接続
```

**段階的移行が可能。Big Bangリライトは不要。**

## 4. 構築ライフサイクル契約（Construction Lifecycle Contract）

### 4.1 3フェーズモデル

```
Phase 1: Skill密結合（PoC）
  ├── 目的：動作する最小限のデモ
  ├── 構造：LLM + Skill関数群（同一プロセス）
  ├── 制約：Skill関数はAPI化可能な形式で書く（4.2参照）
  └── 成果物：ステークホルダーの合意

Phase 2: API分離（商用化準備）
  ├── 目的：コア層の独立
  ├── 構造：Skill関数 → APIエンドポイントとして切り出し
  ├── 制約：API仕様（OpenAPI等）を正式に定義
  └── 成果物：LLMなしで動作するAPI

Phase 3: 疎結合完成（商用）
  ├── 目的：オペレータ層とコア層の完全分離
  ├── 構造：別プロセス/別コンテナ/別サービス
  ├── 制約：オペレータ層はAPI仕様のみに依存
  └── 成果物：交換可能なオペレータ層 + 独立したコア層
```

### 4.2 Skill-to-API ポータビリティ契約

Phase 1 で作るSkill関数が Phase 2 で捨てコードにならないために、
Skill関数は以下の制約を満たさなければならない：

```
Skill関数の契約：
  ├── 引数はプリミティブ型またはシリアライズ可能なオブジェクトのみ
  ├── 戻り値はシリアライズ可能なオブジェクトのみ
  ├── LLMコンテキスト（会話履歴等）に依存しない
  ├── 副作用はDB操作・外部API呼び出しに限定
  └── 関数シグネチャがそのままAPIエンドポイントに変換可能
```

```
✅ ポータブルなSkill関数:
  function(明示的な引数) → 明示的な戻り値
  → そのままAPIのリクエスト/レスポンスに変換可能

❌ ポータブルでないSkill関数:
  function(LLMコンテキスト全体) → 自然言語文字列
  → API化するには全面書き直しが必要
```

## 5. コアAPI設計契約（Core API Design Contract）

### 5.1 API設計原則

コア層のAPIは以下の原則に従う：

| 原則 | 定義 | 理由 |
|------|------|------|
| **リソース指向** | 操作対象をリソースとして定義し、CRUD操作を基本とする | 予測可能なAPI構造 |
| **自己完結** | 1つのAPI呼び出しで1つの操作が完結する | オペレータの実装複雑度を下げる |
| **明確なエラー** | ステータスコード + 人間可読メッセージ + 機械可読コード | LLMがエラーを解釈・対処可能 |
| **冪等性** | 同じリクエストの再送で副作用が増加しない | LLMのリトライ安全性 |
| **自己記述** | レスポンスに次に取り得るアクションを含む | LLMが次の操作を自律的に判断可能 |

### 5.2 単一の真実契約（Single Source of Truth Contract）

API仕様書（OpenAPI等）を唯一の正とし、以下の全てを仕様書から導出する：

```
API仕様書（単一の真実）
  ├── → APIサーバの実装（コード生成 or バリデーション）
  ├── → APIテストの生成
  ├── → オペレータ層のツール定義（Function Calling等）
  ├── → 型安全クライアントの生成
  ├── → UI-API Manifestとの照合（GOVERNANCE.mdのフレームワーク）
  └── → ドキュメントの生成
```

**手書きの二重管理をしない。仕様書と実装の乖離を構造的に排除する。**

### 5.3 LLMフレンドリーレスポンス契約

LLMがAPIの消費者となることを前提に、レスポンスは以下の構造を含むべき：

| 要素 | 目的 | 必須/推奨 |
|------|------|----------|
| **業務データ** | 要求されたリソースの情報 | 必須 |
| **権限フラグ** | 何ができて何ができないかの明示 | 推奨 |
| **利用可能アクション** | 次に呼べるエンドポイントの一覧（HATEOAS的） | 推奨 |
| **人間向け要約** | LLMがユーザーへの説明にそのまま使える文 | 任意 |

**権限判断はコア層が行い、フラグとして返す。**
**オペレータ層（LLM/UI）がビジネスロジックで権限判断することを禁止する。**

### 5.4 エラーハンドリング契約（Error Handling Contract）

#### 5.4.1 コア層のエラー設計

コア層のエラーレスポンスは以下の構造に準拠する：

```
エラーレスポンス構造：
  ├── HTTP ステータスコード（4xx / 5xx）
  ├── 機械可読エラーコード（一意の文字列識別子）
  ├── 人間可読メッセージ（エンドユーザーにそのまま提示可能）
  ├── 詳細情報（フィールドレベルのバリデーションエラー等）
  ├── リトライ可否フラグ（retryable: true/false）
  └── 推奨アクション（次に何をすべきかの提案）
```

#### 5.4.2 エラーの分類とオペレータへの期待

| エラー種別 | コア層の責務 | オペレータ層の責務 |
|-----------|-------------|-------------------|
| **入力エラー（4xx）** | 具体的な理由を返す | ユーザーに分かる形で提示、修正を促す |
| **権限エラー（403）** | 不足権限を明示 | 必要な権限の取得方法を案内 |
| **競合エラー（409）** | 競合の原因を返す | リトライまたは手動解決を案内 |
| **一時障害（5xx）** | retryable フラグを返す | retryable なら自動リトライ |
| **永続障害（5xx）** | retryable: false | 人間にエスカレーション |

#### 5.4.3 オペレータ層のエラー透過原則

```
オペレータ層はコア層のエラーを：
  ├── ✅ 翻訳する（機械向け → 人間/LLM向けの表現に変換）
  ├── ✅ 補足する（「この操作を再試行してください」等の案内を付加）
  ├── ❌ 握りつぶさない（エラーを無視して正常応答を返さない）
  └── ❌ 独自解釈で代替操作しない（エラーを受けて別の操作を勝手に実行しない）
```

### 5.5 認証・認可のレイヤー間契約（AuthN/AuthZ Contract）

#### 5.5.1 認証アーキテクチャ

```
┌──────────────┐     API Key / Token     ┌──────────────┐
│ オペレータ層  │ ──────────────────────→ │  コア層       │
│              │                         │              │
│ 認証の責務:  │                         │ 認証の責務:   │
│ ・エンドユーザ│                         │ ・APIキーの   │
│   ーの本人確認│                         │   検証       │
│ ・セッション  │                         │ ・リクエスト  │
│   管理       │                         │   元の識別   │
└──────────────┘                         └──────────────┘
```

#### 5.5.2 認可の責務分離

| 責務 | 担当 | 理由 |
|------|------|------|
| **ビジネスレベルの認可** | コア層 | 「このユーザーはこの操作ができるか」はビジネスルール |
| **APIアクセス制御** | コア層 | エンドポイントへのアクセス可否 |
| **UIレベルの表示制御** | オペレータ層 | コアが返す権限フラグに基づいてUI要素を制御 |
| **エンドユーザー認証** | オペレータ層（またはIdP） | ログイン画面・セッション管理 |

#### 5.5.3 原則

```
コア層は「誰がリクエストしているか」を知る（APIキー/トークン）
コア層は「その人が何をできるか」を判定する（認可）
コア層は「リクエストがどのUIから来たか」を知らない（オペレータ透過性）

オペレータ層は「エンドユーザーが誰か」を確認する（認証）
オペレータ層は「ビジネス上の可否」を自分で判断しない（コアに委譲）
```

## 6. 可観測性契約（Observability Contract）

### 6.1 コア層の可観測性

コア層は以下の情報を外部から取得可能にしなければならない：

| 観測対象 | 内容 | 目的 |
|---------|------|------|
| **リクエストログ** | 全API呼び出しの入出力とタイムスタンプ | 障害調査・監査 |
| **ヘルスチェック** | コア層の稼働状態を返すエンドポイント | 死活監視 |
| **メトリクス** | レイテンシ、エラー率、スループット | パフォーマンス監視 |
| **監査ログ** | 誰が・いつ・何を・どう変えたか | コンプライアンス |
| **相関ID** | リクエストを追跡するための一意識別子 | 分散トレーシング |

### 6.2 レイヤー間の追跡契約

```
オペレータ層 → コア層：
  ├── リクエストヘッダに相関ID（Correlation-ID）を付与
  ├── オペレータ識別子を付与（どのオペレータからの呼び出しか）
  └── エンドユーザー識別子を付与（誰の操作か）

コア層 → レスポンス：
  ├── 相関IDをレスポンスヘッダに含める
  └── リクエストIDをレスポンスに含める（問い合わせ用）
```

### 6.3 障害時の透明性

```
コア層障害時：
  → オペレータ層はヘルスチェックで検出可能
  → オペレータ層は「コアが応答しません」とユーザーに通知
  → コア層の内部詳細は漏洩させない

オペレータ層障害時：
  → コア層は影響を受けない（障害独立性）
  → 別のオペレータから引き続きAPIアクセス可能
```

## 7. ガードレール契約（Guardrail Contract）

### 7.1 操作リスク分類

オペレータ層がコアAPIを呼び出す際、操作のリスクを分類し、
分類に応じた実行ポリシーを適用する。

| リスクレベル | 分類基準 | 実行ポリシー |
|-------------|---------|-------------|
| **低** | 参照系操作（データ変更なし） | 自動実行 |
| **中** | 単体変更操作（1リソースの作成・更新・削除） | 確認付き実行 |
| **高** | 一括変更・破壊的操作・不可逆操作 | 人間承認必須 |

### 7.2 確認プロトコル

中〜高リスクの操作に対して、オペレータ層は以下のプロトコルに従う：

```
1. 操作の意図を解析し、リスクレベルを判定
2. 影響範囲を算定（影響を受けるリソースの数・種類）
3. 影響範囲を人間に提示し、確認を要求
4. 確認が得られた場合のみ、API呼び出しを実行
5. 実行結果を人間に報告
```

### 7.3 権限境界契約（Permission Boundary Contract）

オペレータ層（LLM含む）に**決して持たせてはならない権限**を定義する：

```
禁止権限：
  ├── 管理者権限の付与・変更
  ├── システム設定の変更
  ├── バックアップ・リストア操作
  ├── 監査ログの削除・改竄
  └── オペレータ自身の設定・権限の変更

実装方法：
  → API認証レベルで制限（オペレータ用APIキーのスコープ）
  → コア層のエンドポイント自体に権限チェックを実装
  → オペレータ層の善意に依存しない
```

**「オペレータ層が守る」のではなく「コア層が拒否する」設計にする。**

## 8. UI多元性パターン（UI Plurality Pattern）

### 8.1 原則

ヘッドレスコアの上には、**複数のオペレータ/UIが同時に存在できる**。
プロダクトチームは「唯一のUI」を作る義務を持たない。

```
ヘッドレスコア
  ├→ LLMオペレータ（自然言語操作）
  ├→ Web UI（管理画面）
  ├→ CLI（自動化）
  ├→ サードパーティUI（顧客が構築）
  ├→ 他システムからのAPI直接呼出
  └→ 将来の未知のインターフェース
```

### 8.2 最小UI原則

プロダクトチームが提供するUIは**最小限**でよい：

| 必須 | 理由 |
|------|------|
| API | コア層そのもの |
| API仕様書 | 全オペレータの構築基盤 |

| 推奨 | 理由 |
|------|------|
| LLMオペレータ | 非エンジニアが使える操作手段 |
| 最小Web UI | 設定・ダッシュボード等のビジュアル操作 |
| CLI | エンジニア・自動化向け（API仕様から自動生成可能） |

| 不要 | 理由 |
|------|------|
| 全業務フローを網羅するUI | APIがあれば顧客が自分で作れる |

### 8.3 UIを作った場合のガバナンス

UIを作った瞬間から、コア層との仕様ズレのリスクが発生する。
UIのガバナンスについては [HEADLESS_ARCHITECTURE_AND_GOVERNANCE.md](./HEADLESS_ARCHITECTURE_AND_GOVERNANCE.md) のフレームワークを適用する。

## 9. アンチパターン集（Anti-Patterns）

### 9.1 設計時のアンチパターン

| アンチパターン | 症状 | 正しいアプローチ |
|---------------|------|-----------------|
| **オペレータ肥大化** | オペレータ層にビジネスロジックが侵食 | 定期的に責務チェック（2.4参照） |
| **API as UI Backend** | 特定UIの画面構造に最適化されたAPI | リソース指向で汎用的に設計 |
| **権限の二重管理** | コア層とオペレータ層の両方で認可ロジック | コア層が権限判断、オペレータはフラグ参照のみ |
| **スキーマなきAPI** | 仕様書なしでAPIを口頭合意で構築 | OpenAPI等を最初から導入 |
| **全部コアに入れる** | UIの表示ロジックまでコア層に実装 | 表示固有ロジックはオペレータ層 |

### 9.2 構築時のアンチパターン

| アンチパターン | 症状 | 正しいアプローチ |
|---------------|------|-----------------|
| **Big Bangリライト** | UI一体型を一度に全部ヘッドレス化 | 3.4の段階的移行 |
| **PoC永住** | Phase 1のSkill密結合のまま商用化 | Phase移行チェック（4.1参照） |
| **仕様書後回し** | 「まず動かしてから仕様書を書く」 | Phase 2開始時に仕様書を正とする |
| **コア層のLLM依存** | コア層内でLLMを呼び出す | コア層は決定論的。LLMはオペレータ層 |

### 9.3 運用時のアンチパターン

| アンチパターン | 症状 | 正しいアプローチ |
|---------------|------|-----------------|
| **オペレータ経由でしかできない操作** | コア層のAPI不足をオペレータがカバー | APIを拡張してコア層の自立性を回復 |
| **監査ログの軽視** | 「後で入れる」 | コア層の初期設計時に組み込む |
| **エラーの握りつぶし** | オペレータがコアのエラーを無視 | エラー透過原則（5.4.3参照） |
| **APIバージョニングなし** | API変更で全オペレータが同時に壊れる | バージョニングで旧版を一定期間維持 |

## 10. 異常系に対する基本方針

### 10.1 可逆性による安全保証

異常系は「起きないようにする」のではなく「起きても戻せる」ことで保証する。

| 対象 | 可逆性の手段 |
|------|-------------|
| コードの変更 | git revert |
| APIの変更 | APIバージョニング |
| データの変更 | 監査ログ + リストア |
| オペレータの判断ミス | 操作の取り消しAPI |
| ガバナンス判断の誤り | ロールバック + 判断基準の校正 |

### 10.2 可逆性を前提としたコア層の設計

コア層は以下を実装すべき：

```
├── 全操作の監査ログ（誰が・いつ・何を・どう変えたか）
├── 論理削除（物理削除の前に復元可能期間を設ける）
├── APIバージョニング（旧版を一定期間維持）
└── バックアップ/リストア手順
```

## 11. フレームワーク適用チェックリスト

具象プロダクトを構築する際、以下の契約が満たされているかを確認する：

### 問題ドメイン認識
- [ ] ヘッドレス化の判断基準（1.2）に2つ以上該当する
- [ ] 既存サービスからの移行の場合、機能要件抽出（1.4）を実施済み

### コア層
- [ ] 決定論性：全APIが同じ入力に対して同じ出力を返す
- [ ] 自立性：オペレータ層なしで単体動作する
- [ ] テスタビリティ：全APIにテストが存在する
- [ ] 監査可能性：全操作の履歴が記録されている
- [ ] 冪等性：同一リクエストの再送で副作用が増加しない

### オペレータ層
- [ ] 翻訳者性：ビジネスロジックを含まない
- [ ] 交換可能性：別のオペレータに置換可能
- [ ] 非保持：業務データを自身に保持しない
- [ ] 障害独立性：停止してもコア層に影響しない

### 責務分割
- [ ] 責務分割ヒューリスティクス（2.4）に基づき全処理を分類済み
- [ ] グレーゾーンの処理はコア層に寄せている

### API設計
- [ ] 単一の真実：API仕様書が全ての導出元になっている
- [ ] リソース指向：CRUDベースの予測可能な構造
- [ ] 権限フラグ：権限判断はコア層が行い、フラグで返す
- [ ] エラーハンドリング：全エラーが5.4の構造に準拠

### 認証・認可
- [ ] コア層がビジネスレベル認可を担当
- [ ] オペレータ層がエンドユーザー認証を担当
- [ ] APIアクセス制御がコア層に実装されている

### 可観測性
- [ ] 全API呼び出しのリクエストログ
- [ ] ヘルスチェックエンドポイント
- [ ] 相関IDによるリクエスト追跡
- [ ] 監査ログ（誰が・いつ・何を・どう変えたか）

### ガードレール
- [ ] リスク分類：全操作がリスクレベルに分類されている
- [ ] 権限境界：禁止権限がAPIレベルで制限されている
- [ ] 確認プロトコル：中〜高リスク操作に確認フローがある

### 可逆性
- [ ] 全操作の監査ログ
- [ ] 論理削除の実装
- [ ] APIバージョニング
- [ ] バックアップ/リストア手順

### アンチパターン確認
- [ ] セクション9のアンチパターンに該当するものがない
