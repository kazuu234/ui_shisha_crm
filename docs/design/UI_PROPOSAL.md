# 企画書: UI Shisha CRM

> コア層企画書: `docs/reference/PROPOSAL_SHISHA_CRM.md`
> コア層基本設計書: `docs/reference/BASIC_DESIGN.md`
> 本書はコア層（Headless Shisha CRM）の Service 層を利用するオペレータ層（UI）の企画書である。

## 1. プロジェクト概要

### 背景

シーシャ屋は顧客との関係性がビジネスの核であり、常連化が売上に直結する。しかし現状は顧客管理の仕組みがなく、以下の問題を抱えている。

| # | 課題 | 現状 |
|---|------|------|
| 1 | **顧客セグメントが不在** | 新規・リピート・常連の区別ができず、接客の質にバラつきがある |
| 2 | **顧客属性の未把握** | 年齢・居住地・シーシャ歴などの基本情報が記録されていない |
| 3 | **顧客情報の属人化** | スタッフ間で情報が共有されず、担当が変わると関係性がリセットされる |
| 4 | **接客の継続性がない** | 会話内容や「次に聞くべきこと」が記録されず、前回の続きから話せない |
| 5 | **会計データの孤立** | Airレジのデータが顧客行動と紐づいておらず、分析に使えない |
| 6 | **来店推移が不可視** | 誰がどの頻度で来店し、リピートしてくれたかのヒストリーが分からない |

現在の運用は **Airレジの明細名にお客様の名前を書いているだけ** であり、顧客管理と呼べるものは存在しない。

コア層（Headless Shisha CRM）は、これらの課題を解決するビジネスロジック・データ管理基盤を提供する。しかしロジックだけでは接客の現場で使えない。**本プロジェクト（UI Shisha CRM）は、コア層の Service 層・Model 層を利用して接客現場で使える画面を提供するオペレータ層**である。

### コア層との関係

#### アーキテクチャ概要

コア層が全てのビジネスロジック・データ管理を担い、UI はそれを人間が操作できる画面に翻訳する。

| 層 | リポジトリ | 責務 |
|---|---|---|
| **コア層** | `headless_shisha_crm` | ビジネスロジック・データ管理・外部連携。Service 層・Model 層として実装。REST API としても公開 |
| **オペレータ層（UI）** | `ui_shisha_crm`（本プロジェクト） | コア層の Service 層を呼び出し、人間が操作できる画面を提供する |

#### ランタイム構成: UI app 組み込み方式

UI は独立した Django app として開発し、**ランタイムではコア層の Django プロジェクトに組み込んで同一プロセスで動作する**。

```
headless_shisha_crm/              # コア層 Django プロジェクト
  config/urls.py                  # /api/v1/... (REST API) と /s/... /o/... (UI) が共存
  customers/services.py           # ビジネスロジック（コア層）
  visits/services.py              #
  ...                             #
  ui/                             # ← ui_shisha_crm から配置される UI app
    staff/views.py                # Service 層を直接呼び出してテンプレートをレンダリング
    owner/views.py                #
    templates/                    #
    static/                       #
```

**この方式を採用する理由:**

| 観点 | 判断 |
|------|------|
| **通信経路** | UI View → Service 層を直接呼び出し。HTTP 越しの自己呼び出しは過剰設計 |
| **認証** | Django の SessionAuthentication がそのまま機能する。セッション転送の仕組みが不要 |
| **Django らしさ** | 標準的な Django アプリケーション構成。Form → View → Service の自然な流れ |
| **キャッシュ問題の回避** | サーバーサイドで HTML を組み立てて返す。React/SPA のような JS バンドルのキャッシュ問題が構造的に発生しない |

**ヘッドレスアーキテクチャとの整合性:**

ヘッドレスの原則は「UI にビジネスロジックを持たせない」「UI を差し替え可能にする」ことであり、「必ず HTTP API 経由で通信する」ことではない。本構成では:

- **ビジネスロジックの分離は維持する** — UI View は Service 層を呼ぶだけ。判定・計算・データ操作は Service/Model 層の責務
- **REST API は引き続き公開される** — 将来 LINE Bot や CLI が同じ API を利用できる。UI が API を潰すわけではない
- **UI の差し替え可能性は維持する** — UI app を外しても Service 層・API は影響を受けない
- **通信経路のみ妥協する** — 「API 経由」ではなく「Service 層への直接呼び出し」。これは同一 Django プロジェクト内の合理的な選択

> ヘッドレスアーキテクチャの詳細: `docs/framework/HEADLESS_PRODUCT_DESIGN_GUIDE.md`
> UI-API 契約管理・変更ガバナンス: `docs/framework/HEADLESS_ARCHITECTURE_AND_GOVERNANCE.md`

#### コア層の構造

コア層は Django + Django REST Framework で構築され、8 app・26 エンドポイントを持つ。UI は主に各 app の Service 層を利用する。

| App | 責務 | UI が利用する Service / Model |
|-----|------|-------------------------------|
| `core` | 共通基盤（Manager, Mixin, Permission, エラー標準, 監査, 冪等性, 可観測性） | `BaseStoreViewSet` のパターンを UI View でも踏襲。`StoreScopedManager`, `SoftDeleteManager` |
| `tenants` | StoreGroup, Store。マルチテナント構造 | `Store` モデル（`request.user.store` でスコープ取得） |
| `accounts` | Staff（カスタム User）、QR 認証 | `QRAuthService`, `Staff` モデル |
| `customers` | 顧客 CRUD | `CustomerService`, `Customer` モデル |
| `visits` | 来店記録、セグメント判定、閾値管理 | `VisitService`, `SegmentService`, `Visit` / `SegmentThreshold` モデル |
| `tasks` | ヒアリングタスク生成・auto close・表示 | `HearingTaskService`, `HearingTask` モデル |
| `imports` | Airレジ CSV インポート・マッチング | `ImportService`, `MatchingService`, `CsvImport` / `CsvImportRow` モデル |
| `analytics` | 分析（日別来客、セグメント比率、スタッフ別対応数） | `AnalyticsService` |

> API エンドポイント全一覧: `docs/reference/BASIC_DESIGN.md` セクション 5
> モデル定義: 同 セクション 3

#### 認証・認可

| 項目 | 方式 |
|------|------|
| **認証** | `SessionAuthentication`。QR トークンでログイン → セッション Cookie で維持。UI とAPI で認証基盤を共有 |
| **認可** | ロールベース。`owner`（全機能）/ `staff`（記録系のみ）の 2 ロール |
| **CSRF** | Django テンプレートの `{% csrf_token %}` で注入。標準的な Django フォーム送信 |
| **権限の表示制御** | `request.user.role` に基づいて UI のメニュー・ボタンを出し分け。権限の実際の強制は Service 層の Permission チェックが行う |

#### Store スコープ

全業務データは `Store`（店舗）に紐づく。コア層の `StoreScopedManager` / `SoftDeleteManager` が `request.user.store` で自動スコープし、他店舗のデータは一切見えない。UI View は `request.user.store` を Service に渡すだけでよい。MVP では単一店舗運用。

### オペレータ層の責務境界

オペレータ層は**翻訳者**である。Service 層を呼び出して結果を画面に変換するだけで、ビジネスロジックを自身に持たない。

#### 基本原則

| 原則 | 内容 | 具体例 |
|------|------|--------|
| **ビジネスロジックを持たない** | 権限チェック・バリデーション・セグメント判定等は全て Service / Model 層が行う | セグメントの色分け表示は UI の仕事。セグメントの判定ロジックは `SegmentService` の仕事 |
| **read は Manager 直接参照可、write は Service 必須** | データ取得（read）はコア層の Manager / QuerySet を直接利用してよい。データ変更（create/update/delete）は必ず Service 層を経由する。Manager はスコープ制御（`for_store`, `alive`）を担い、Service は業務ルールの検証・副作用の実行を担う | read: `Customer.objects.for_store(store)` は OK。write: `Customer.objects.create(...)` は NG → `CustomerService.create_customer(...)` を使う |
| **UI を外してもコア層は影響を受けない** | UI app を Django プロジェクトから除去しても、Service 層・REST API は単体で動作する | 将来 LINE Bot や CLI が REST API を利用する際、UI の有無は無関係 |

#### 例外: 補助的ロジック

UX 上の表示変換・入力補助・クライアントサイドの一時状態管理など、オペレータ層固有のロジックは存在しうる。

| 補助ロジックの例 | UI 側で行う理由 |
|-----------------|----------------|
| フォームの入力リアルタイムバリデーション（文字数カウント等） | UX のためだが、最終的な検証は Service 層が行う |
| 検索キーワードのデバウンス | サーバーへのリクエスト頻度の最適化 |
| 画面遷移時の一時的なフォーム状態保持 | UX のため（API には未送信） |
| セグメント名に応じたバッジの色分け | 表示の問題であり、ビジネスロジックではない |

これらの追加・変更は、変更レベル分類（Level 0〜3）に基づいて影響範囲を判断し、設計レビューを経て実施する。

> 変更レベル分類の定義: `docs/framework/HEADLESS_ARCHITECTURE_AND_GOVERNANCE.md` セクション 3
> 将来的には MAGI 分権判断機構（同 セクション 4）による自動化を想定するが、MVP 時点では人間によるレビューで代替する

### 2 系統の UI

本プロジェクトは利用デバイスと業務特性の違いから、**2 系統の UI** を構築する。

#### なぜ 2 系統に分けるか

スタッフとオーナーでは、デバイス・利用タイミング・求める情報量・操作スタイルが根本的に異なる。1 つの UI で両方を満たそうとすると、スタッフにとっては情報過多で操作が遅くなり、オーナーにとっては情報が足りず分析業務が非効率になる。

#### スタッフ UI（タブレット）

| 項目 | 内容 |
|------|------|
| **デバイス** | タブレット（店舗に 1 台共有） |
| **利用者** | 全スタッフ（オーナー含む。接客時はスタッフ扱い） |
| **利用タイミング** | 来店時〜接客中〜会計後 |
| **設計哲学** | **タッチ 1〜2 回で事柄が進む**。情報量を絞り、今やるべきことだけを表示する |
| **入力方式** | タッチ（選択・タップ）が主。文字入力は例外的（会話メモ等） |
| **表示する情報** | 顧客の基本情報・簡易編集、ヒアリングタスク、直近の来店履歴・簡易表示、会話メモ、会計後の簡易マッチング |
| **表示しない情報** | 分析データ、セグメント閾値設定、スタッフ管理、CSV アップロード・一覧精査 |

#### オーナー UI（PC）

| 項目 | 内容 |
|------|------|
| **デバイス** | PC（オーナー個人） |
| **利用者** | オーナーのみ |
| **利用タイミング** | 閉店後の振り返り、営業分析、スタッフ管理、Airレジ連携 |
| **設計哲学** | **分析業務を効率的に行える情報密度**。表示量の制約なし |
| **入力方式** | キーボード + マウス |
| **表示する情報** | 分析ダッシュボード、顧客一覧（詳細フィルタ・ソート）、来店推移、売上連動分析、スタッフ管理、セグメント閾値設定、CSV インポート・マッチング |

### MVP スコープ

コア層の企画書で定義された MVP 機能に対応する UI を構築する。API + UI セットでリリースする。

#### 機能と UI 系統の対応

| # | 機能 | 概要 | スタッフ UI | オーナー UI |
|---|------|------|:-----------:|:-----------:|
| F-01 | **顧客マスタ管理** | 名前・年齢・居住エリア・シーシャ歴・LINE ID・メモ。CRUD | o（登録・簡易表示・基本編集） | o（一覧・詳細・フル編集） |
| F-02 | **来店記録** | 日付 + 顧客 + 対応スタッフ + 会話メモ | o（作成・簡易表示） | o（一覧・編集・削除） |
| F-05 | **顧客セグメント** | 来店回数ベースの自動分類。閾値設定 | o（表示のみ） | o（表示 + 閾値設定） |
| F-06 | **スタッフ管理** | オーナー/レギュラー/テンポラリの管理、QR 発行 | - | o |
| F-07 | **ヒアリングタスク** | 未取得属性の自動タスク生成・表示・auto close | o（表示・消化） | o（一覧確認） |
| F-08 | **権限制御** | ロール別の画面・機能アクセス制御 | o（Service 層が制御） | o（Service 層が制御） |
| F-09 | **Airレジ CSV インポート** | CSV アップロード・マッチング・確定 | o（会計後の簡易マッチング） | o（CSV アップロード・一覧精査・マッチング管理） |
| F-11 | **来客分析** | 日別来客数・セグメント比率・スタッフ別対応数 | - | o |

**凡例**: o = 対応する画面を持つ、- = この系統では提供しない

**注意**: 同じ機能がスタッフ UI・オーナー UI の両方に存在する場合でも、画面は完全に別実装である。PC 版をレスポンシブ対応するのではなく、タブレット専用の UI として独立して設計・実装する。

> MVP 機能定義の詳細: `docs/reference/PROPOSAL_SHISHA_CRM.md` セクション 3, 4

## 2. ユーザーと利用シーン

### ユーザー種別

| 種別 | 人数 | UI 系統 | 主な操作 |
|------|------|---------|----------|
| **オーナー（兼スタッフ）** | 1 名 | PC（管理・分析）+ タブレット（接客時） | 分析ダッシュボード、スタッフ管理、セグメント閾値設定、CSV インポート管理。接客時はスタッフと同等 |
| **レギュラースタッフ** | 約 5 名 | タブレット | 顧客選択・新規登録、来店記録作成、ヒアリングタスク消化、会話メモ、会計後マッチング |
| **テンポラリスタッフ** | 不定 | タブレット | レギュラーと同等の操作。有名なシーシャの作り手が立つことがあり、その人にお客がつく構造 |

レギュラーとテンポラリは権限上は同一の「スタッフ」ロール。種別の違いはスタッフマスタの属性として管理し、UI の操作範囲には影響しない。

### タブレット共有運用

- 店舗にタブレット **1 台を全スタッフで共有** する
- シフト交代時にログイン切替を行う（前のスタッフがログアウト → 次のスタッフがログイン）
- **ログイン QR はスタッフ個別**。`request.user` が操作者本人を表す。来店記録の `staff` フィールドには `request.user` が紐づき、操作者の監査性を確保する
- 接客中に別のスタッフがタブレットを使う場合は、前のスタッフがログアウト → 次のスタッフが QR で再ログインする。MVP ではシフト中は基本的に 1 人が操作する前提とし、簡易切替 UI は Phase 2 で検討する（未決事項 D-02 参照）

### 利用シーン

#### シーン 1: 来店時（スタッフ UI / タブレット）

お客さんが来店し、スタッフが最初にタブレットで行う操作。

```
お客さんが来店
  │
  ├── 「来たことありますか？」と会話で確認
  │
  ├── 既存顧客の場合
  │     └── 顧客を選択する
  │           ├── 名前で検索
  │           └── 最近来た順の一覧から選ぶ
  │
  └── 新規顧客の場合
        └── 新規登録（名前は必須。他は後から埋める）
  │
  ▼
顧客を選択した時点で:
  ├── ヒアリングタスク（未取得の属性）が表示される
  │     例: 「年齢を聞いてください」「居住エリアを聞いてください」
  ├── 直近の来店履歴・会話メモが表示される
  └── 顧客のセグメント（新規/リピート/常連）が表示される
  │
  ▼
接客開始
```

**ポイント**: 顧客を選んだ瞬間に「この人に何を聞くべきか」「前回何を話したか」が分かる。これが接客の継続性を実現する。

#### シーン 2: 接客中（スタッフ UI / タブレット）

接客しながらタブレットを操作する。リアルタイムでの記録が基本。

| 操作 | 入力方式 | 頻度 |
|------|---------|------|
| ヒアリングタスク消化（年齢・居住エリア・シーシャ歴の入力） | 選択式タップ（数値ピッカー、プリセット選択） | 新規顧客の初回〜数回 |
| 会話メモの記入 | テキスト入力（例外的。短文） | 必要に応じて |
| 来店記録の作成 | タップ（顧客 + スタッフは選択済み。日付は当日デフォルト） | 来店ごとに 1 回 |

**制約**: タブレットでの文字入力は最小限に抑える。ヒアリング項目は選択式にし、自由入力は会話メモのみ。

#### シーン 3: 会計後（スタッフ UI / タブレット）

会計が終わった後、可能であれば Airレジの明細と CRM の来店記録を紐づける。

```
会計完了（Airレジで会計処理済み）
  │
  ├── すぐにマッチングできる場合
  │     └── Airレジ明細と来店記録を紐づけ確定
  │
  └── 忙しくてすぐにできない場合
        └── マッチング未実施のまま残る（浮遊売上）
              ※ 浮遊しても「その日の店舗売上」としてはデータが残るため、
                 日別売上の把握には支障がない。
                 顧客別の売上貢献度の分析精度が落ちる、という影響。
```

**課題**: CSV のマッチングは結合キー（顧客名の部分一致等）で行うが、確実にマッチできるとは限らない。マッチング操作を誰がいつやるかの運用設計が重要。理想は会計直後にスタッフが行うこと。

#### シーン 4: 閉店後の振り返り（オーナー UI / PC）

オーナーがその日の営業を振り返る。

| 確認する情報 | 利用する Service | 内容 |
|-------------|-----------------|------|
| 日別来客数 | `AnalyticsService.daily_summary()` | 今日は何人来たか |
| 新規/リピート/常連の比率 | `AnalyticsService.segment_ratio()` | 新規獲得できたか、常連は来てくれたか |
| スタッフ別対応数 | `AnalyticsService.staff_summary()` | 誰がどれだけ接客したか |
| 来店推移 | `AnalyticsService.daily_summary()`（期間指定） | 週次・月次の来客トレンド |

**今後の拡張候補**（Phase 2 以降）: 売上金額の推移、顧客別売上貢献度、フレーバー消費傾向。これらは API 側の拡張と合わせて検討する。

#### シーン 5: 管理業務（オーナー UI / PC）

オーナーが行う管理・設定業務。頻度は低いが、正確な操作が求められる。

| 業務 | 頻度 | 操作内容 |
|------|------|---------|
| スタッフ管理 | 月数回 | スタッフ追加・無効化、QR コード発行 |
| セグメント閾値設定 | 月 1 回程度 | 新規/リピート/常連の閾値変更。変更すると全顧客のセグメントが即時再計算される |
| CSV インポート | 営業日ごと | Airレジ CSV のアップロード。バリデーション結果確認。マッチング候補の確認・確定・却下 |
| 顧客データ管理 | 随時 | 顧客情報の詳細編集、来店履歴の閲覧・修正（日付変更・論理削除等） |

### 顧客の同定方法

システムによる自動同定・重複検知は MVP 外。スタッフが会話で「来たことありますか？」と確認する運用。

UI は以下の 2 つの手段で顧客を探せるようにする:

1. **名前検索** — 名前の一部を入力して絞り込む
2. **最近来た順の一覧** — 直近に来店した顧客が上に表示される。常連は高頻度で表示されるため、素早く選べる

両方を併用できること。検索結果にもセグメント（新規/リピート/常連）を表示し、視覚的に識別できるようにする。

## 3. UI 方針・制約

### 2 系統の分離方針

スタッフ UI（タブレット）とオーナー UI（PC）は **完全に分離** する。レスポンシブで兼用しない。

| 判断 | 理由 |
|------|------|
| テンプレートを分離する | デバイス・操作体系・情報密度が根本的に異なる。レスポンシブで兼用すると両方が中途半端になる |
| URL 名前空間を分離する | 例: スタッフ UI は `/s/`、オーナー UI は `/o/`。ルーティングレベルで完全に別世界 |
| スタッフ画面を PC で使うことはない | タブレット専用として最適化してよい |
| オーナー画面をタブレットで使うことはない | PC の情報密度を前提に設計してよい |

### スタッフ UI の設計原則

#### 原則 1: 1 項目につきタッチ 1〜2 回で完了する

全ての操作を「タップで進む」に統一する。1 つの入力項目（例: 年齢の入力、顧客の選択）を完了するのにタッチ 1〜2 回以内。3 回以上のタッチが必要な操作は設計を見直す。

| 良い例 | 悪い例 |
|--------|--------|
| シーシャ歴: 「なし / 初心者 / 中級 / 上級」の 4 択をタップ | シーシャ歴: テキストフィールドに自由入力 |
| 顧客選択: 一覧からタップ | 顧客選択: 名前を全文字入力して検索してから選択 |
| 来店記録作成: 顧客・スタッフ選択済み → 「記録」ボタン 1 タップ | 来店記録: 日付ピッカー → 顧客選択 → スタッフ選択 → 確認画面 → 確定 |

#### 原則 2: 詰め込まず、かつスクロールさせない

「情報量を絞る」は「画面の項目数を極端に減らす」ことではない。必要な情報を 1 画面（スクロールなし）に収めることが目標。

| 良い例 | 悪い例 |
|--------|--------|
| 顧客選択後: 名前・セグメント・タスク・直近来店が 1 画面に収まる | 顧客選択後: 名前だけ表示。タスクを見るには別画面に遷移 |
| ヒアリングタスク 3 件がカード形式で並ぶ | タスクが 1 件ずつ表示され、スワイプで切り替え |

**判断基準**: 「接客に必要な情報が、スクロールせずに一目で分かるか？」

#### 原則 3: 文字入力は例外

タブレットでのキーボード入力は避ける。文字入力が必要なのは以下のみ:

| 文字入力が必要な場面 | 理由 |
|---------------------|------|
| 顧客の名前（新規登録時） | 必須フィールド。代替手段がない |
| 会話メモ | 自由記述が本質的に必要。ただし短文を想定 |
| 顧客検索（名前の一部入力） | 一覧から選べない場合の補助手段 |

それ以外の入力（年齢、居住エリア、シーシャ歴等）は全て **選択式** にする。

#### 原則 4: ナビゲーションはボトムタブ

タブレットでは画面下部のタブで主要機能を切り替える。サイドバーは使わない（タブレットでは操作しづらく、画面を圧迫する）。

タブの構成案（基本設計で確定）:
- 顧客（選択・新規登録）
- 接客中（タスク・メモ）
- 来店記録
- （必要に応じて追加）

### オーナー UI の設計原則

#### 原則 1: 情報密度を優先する

PC の広い画面を活かし、一覧性と操作効率を重視する。テーブル表示、フィルタ・ソートの充実、一括操作。

| 画面 | 情報密度の例 |
|------|-------------|
| 顧客一覧 | 名前・セグメント・来店回数・最終来店日・未消化タスク数を 1 行に表示。列ソート・セグメントフィルタ |
| 分析ダッシュボード | 日別来客グラフ・セグメント比率円グラフ・スタッフ別棒グラフを同一画面に配置 |
| CSV インポート | アップロード状態、行一覧、マッチング候補を段階的に展開表示 |

#### 原則 2: 管理業務は正確性を重視する

スタッフ管理・閾値設定・CSV マッチングなどは頻度が低いが影響が大きい操作。確認ステップや取消手段を設ける。

| 操作 | 確認手段 |
|------|---------|
| スタッフ無効化 | 「このスタッフを無効化しますか？」の確認ダイアログ |
| セグメント閾値変更 | 変更プレビュー（「この変更で N 件の顧客のセグメントが変わります」）|
| 来店記録の論理削除 | 「削除すると来店回数とセグメントが再計算されます」の警告 |

#### 原則 3: ナビゲーションはサイドバー

PC では左サイドバーで主要機能を常に表示する。階層的なメニュー構造に適している。

サイドバーの構成案（基本設計で確定）:
- ダッシュボード（分析）
- 顧客管理
- 来店記録
- スタッフ管理
- セグメント設定
- Airレジ連携
- （必要に応じて追加）

### フォーム設計原則

全画面共通のフォーム設計ルール。

| 原則 | 内容 |
|------|------|
| **選択式優先** | 選択肢が定義できるフィールドは必ず選択式にする。テキスト入力にしない |
| **デフォルト値の活用** | 来店日 = 今日、対応スタッフ = ログイン中のスタッフ、など。入力の手間を減らす |
| **必須/任意の明示** | 必須フィールドと任意フィールドを視覚的に区別する |
| **段階的入力** | 新規顧客登録時、名前だけで登録可能。他の項目はヒアリングタスクとして後から埋める |
| **バリデーション二重構造** | クライアントサイドで即時フィードバック（文字数超過等）。最終検証は Service 層が行う |

### フィードバック・エラー表示

| 種別 | 表示方法 |
|------|---------|
| **操作成功** | トースト通知（数秒で自動消去）。「来店記録を作成しました」等 |
| **バリデーションエラー** | フィールド横にインライン表示。Service 層が raise する ValidationError のメッセージをそのまま表示 |
| **権限エラー（403）** | 「この操作にはオーナー権限が必要です」。操作不可のボタンはそもそも非表示にする |
| **サーバーエラー（500）** | 「エラーが発生しました。しばらくしてから再度お試しください」。ユーザーを不安にさせない |
| **ネットワークエラー** | 「サーバーに接続できません」。リトライボタンを表示 |

### ローディング・非同期操作

| 場面 | 方針 |
|------|------|
| **画面遷移時** | ページ全体のローディングインジケーター |
| **データ送信時**（フォーム submit） | ボタンを無効化 + スピナー表示。二重送信を防止 |
| **一覧の読み込み** | スケルトンスクリーンまたはスピナー。空状態とローディング状態を区別する |
| **重い処理**（CSV インポート、一括再計算） | プログレス表示。ブラウザを閉じても処理は API 側で継続する旨を表示 |

### アクセシビリティ（MVP 最低限）

| 対応項目 | 内容 |
|---------|------|
| **コントラスト比** | WCAG 2.1 AA 準拠（4.5:1 以上） |
| **タッチターゲット** | スタッフ UI のボタン・タップ領域は最低 44x44px |
| **フォーカス管理** | PC のオーナー UI でキーボードナビゲーションが機能する |
| **エラーの色以外の識別** | エラー表示は赤色だけでなく、アイコン + テキストでも識別可能に |

### 権限による表示制御

`request.user.role` に基づいて UI の表示を制御する。

| 制御方針 | 内容 |
|---------|------|
| **スタッフ UI は全員スタッフ扱い** | スタッフ UI（`/s/...`）ではロールに関わらず同一の表示・操作範囲。オーナーがタブレットで接客する際もスタッフと同じ画面を使う。オーナー固有の機能（編集・削除等）はオーナー UI（`/o/...`）でのみ提供する |
| **オーナー UI はオーナーのみ** | オーナー UI（`/o/...`）にスタッフロールでアクセスした場合はリダイレクト（403 ページではなくスタッフ UI のホームに戻す） |
| **URL 直接アクセスの防御** | 権限のない URL にアクセスした場合はリダイレクト |
| **Service 層が最終防御** | UI の表示制御はあくまで UX。権限の実際の強制は Service 層の Permission チェックが行う |

## 4. 技術方針

### なぜ Django テンプレートか

| 選択肢 | 採用 | 理由 |
|--------|------|------|
| **Django テンプレート（SSR）** | **採用** | サーバーサイドで HTML を組み立てて返す。コア層と同一 Django プロジェクトに同居するため、Session 認証・CSRF・テンプレートタグが自然に機能する。キャッシュ問題が構造的に発生しない |
| React / Next.js 等の SPA | 不採用 | JS バンドルのブラウザキャッシュが原因で、再リリースしても更新されない問題が頻発する。MVP の規模に対して過剰。Django との二重構成になりメンテコストが増大する |
| Vue / Svelte 等の軽量 SPA | 不採用 | React と同様のキャッシュ問題。ビルドパイプラインが必要になり、Django テンプレートの簡素さが失われる |

**判断の核心**: ブラウザに状態を持たせない。サーバーが正しい HTML を返せば画面は常に最新。デプロイ = 即反映。

### UI View の設計方針

UI View はコア層の Service 層を呼び出して結果をテンプレートに渡す。ビジネスロジックを View に書かない。

#### 基本パターン

```python
# ui/staff/views/customer.py（イメージ）
class CustomerSelectView(LoginRequiredMixin, StaffRequiredMixin, TemplateView):
    """顧客選択画面"""
    template_name = "ui/staff/customer_select.html"

    def get_context_data(self, **kwargs):
        store = self.request.user.store
        # Service / Manager 経由でデータ取得
        customers = Customer.objects.for_store(store).order_by('-last_visited_at')
        return {**super().get_context_data(**kwargs), "customers": customers}
```

```python
# ui/staff/views/visit.py（イメージ）
class VisitCreateView(LoginRequiredMixin, StaffRequiredMixin, CreateView):
    """来店記録作成"""
    form_class = VisitCreateForm
    template_name = "ui/staff/visit_create.html"

    def form_valid(self, form):
        store = self.request.user.store
        # Service 層に委譲。View はロジックを持たない
        visit = VisitService.create_visit(
            store=store,
            customer=form.cleaned_data['customer'],
            staff=self.request.user,
            visited_at=form.cleaned_data['visited_at'],
            conversation_memo=form.cleaned_data.get('conversation_memo', ''),
        )
        return redirect('staff:visit_detail', pk=visit.pk)
```

#### 守るべきルール

| ルール | 内容 |
|--------|------|
| **View は薄く** | データ取得は Manager、データ操作は Service 層。View は「取得して渡す」「受け取って Service に渡す」のみ |
| **Model を直接 create/update しない** | `Customer.objects.create(...)` を View に書かない。Service 層を経由する |
| **QuerySet の構築はコア層に従う** | `for_store(store)` や `alive()` はコア層の Manager が提供する。UI View で `.filter(is_deleted=False)` を直書きしない |
| **権限チェックは Mixin + Service 層** | `LoginRequiredMixin`, `StaffRequiredMixin`, `OwnerRequiredMixin` で View レベルのガード。業務ルールの権限チェックは Service 層 |
| **Form はバリデーションに専念** | Django Form はフィールド定義と入力バリデーション。ビジネスルールの検証は Service 層 |

### CSS フレームワーク選定

MVP 時点では **未確定**。基本設計の画面設計フェーズでデザインの方向性が固まった段階で選定する。

#### 選定基準

| 基準 | 重要度 | 内容 |
|------|--------|------|
| **タブレットタッチ UI との相性** | 高 | タッチターゲットのサイズ、モバイルファーストのコンポーネントが揃っているか |
| **Django テンプレートとの親和性** | 高 | ビルドステップなしで使えるか、CDN 配信で即利用可能か |
| **2 系統の分離に適しているか** | 中 | スタッフ UI とオーナー UI で異なるスタイルを適用しやすいか |
| **学習コスト** | 低 | AI 実装（Cursor）が扱いやすいか |

#### 候補

| フレームワーク | 特徴 | 適合度 |
|---------------|------|--------|
| **Tailwind CSS** | ユーティリティファースト。テンプレート内でスタイルが完結する。ビルドステップが必要（ただし CDN 版あり） | デザインの自由度が高い。ただし MVP ではビルドパイプラインをどうするかが論点 |
| **Bootstrap 5** | コンポーネントベース。CDN で即利用可能。タッチ対応のコンポーネントが豊富 | 導入が容易。見た目の差別化はしにくいが MVP では十分 |
| **daisyUI（Tailwind 拡張）** | Tailwind ベースにコンポーネントクラスを追加。Tailwind の自由度 + Bootstrap 的な手軽さ | 両方の良さを取れるが、Tailwind のビルドステップは必要 |

> **決定時期**: UI 基本設計書の画面設計フェーズで決定する

### JavaScript の方針

**サーバーサイドレンダリング（SSR）が主軸**。画面遷移と主要なデータ操作は Django View が HTML を返す形で処理する。JavaScript はページ内のインタラクティブな操作に使う。

#### SSR と JS の役割分担

| 処理 | 担当 | 手段 |
|------|------|------|
| **画面遷移** | SSR | Django View がテンプレートを返す。通常のリンク・フォーム送信 |
| **データの作成・更新・削除** | SSR | Django Form → View → Service 層。POST/PUT でサーバーに送信 |
| **一覧の読み込み・フィルタ・ソート** | SSR | Django View がフィルタ済みの QuerySet をテンプレートに渡す |
| **部分的なページ更新** | JS（HTMX） | HTMX が Django View を呼び、HTML フラグメントを返してページ内を差し替え。例: 顧客検索のインクリメンタル絞り込み |
| **小さな UI インタラクション** | JS | ドロップダウンの開閉、モーダル表示、トースト通知、フォーム入力のリアルタイムフィードバック |
| **複雑なクライアント状態管理** | 使わない | SPA 的な状態管理（Redux 等）は不要。状態はサーバーが持つ |

#### HTMX の採用

ページ全体をリロードせずに部分更新が必要な場面で **HTMX** を使う。

| HTMX の利用場面 | 内容 |
|----------------|------|
| 顧客検索 | 名前の入力に応じて候補一覧を部分更新 |
| ヒアリングタスク消化 | タスクをタップして完了 → そのタスクだけ画面から消える |
| マッチング候補選択 | 候補を選択 → 確定結果が即反映 |
| フォーム送信後のフィードバック | 送信 → サーバーが結果の HTML フラグメントを返す → トースト表示 |

**HTMX が呼ぶ先は Django View**（HTML フラグメントを返す）であり、REST API を直接呼ぶのではない。これは UI app 組み込み方式（セクション 1）と一貫している。

#### その他の JS ライブラリ

| ライブラリ | 用途 | 採否 |
|-----------|------|------|
| **Alpine.js** | HTMX では対応しきれない小さなクライアントサイド状態管理（タブ切替、ドロップダウン等） | 検討中。必要性が出た段階で導入判断 |
| **Chart.js / ApexCharts** | オーナー UI の分析グラフ描画 | 採用見込み。分析ダッシュボードにグラフは必須 |
| **Vanilla JS** | HTMX・Alpine.js でカバーできない個別処理 | 必要に応じて。ライブラリを増やすより素の JS で書く |

### 静的ファイル管理

| 項目 | 方針 |
|------|------|
| **配信方法** | Django の `collectstatic` で静的ファイルを収集。本番では Nginx 等で配信 |
| **CSS** | CSS フレームワーク選定結果に依存。Tailwind を採用する場合はビルドステップが必要（`npx tailwindcss`）。Bootstrap の場合は CDN or vendored で即利用 |
| **JS** | バンドラー（webpack 等）は使わない。HTMX・Chart.js 等は CDN または vendored ファイルとして `static/` に配置。`<script>` タグで読み込む |
| **画像** | `static/ui/images/` に配置。ユーザーアップロード画像は MVP では存在しない |

### テスト方針

| テスト種別 | ツール | 対象 | 方針 |
|-----------|--------|------|------|
| **View テスト** | Django TestClient (`pytest-django`) | UI View のレスポンスステータス、テンプレート選択、コンテキストデータ | 全 View に対して正常系 + 権限エラーをカバー |
| **Form テスト** | pytest | Django Form のバリデーション | 必須フィールド、選択肢、エッジケース |
| **E2E テスト** | 候補: Playwright | 主要な業務フロー（来店記録作成、顧客登録等） | ツール選定・対象範囲は未決（D-04 参照）。MVP では Django TestClient を主軸とし、E2E の要否は基本設計完了時に判断する |
| **ビジュアルリグレッション** | MVP 外 | - | Phase 2 で検討 |

**テストの責務境界**: Service 層のビジネスロジックテストはコア層の責務。UI テストは「正しい Service が呼ばれて、正しいテンプレートが返されるか」を検証する。

**モック vs 実 DB**: UI テストは **実 DB + 実 Service で動作させる**（モックしない）。理由:
- UI はコア層と同一 Django プロジェクトで動作するため、テスト時もそのまま実 Service が使える
- Service 層をモックすると、シグネチャ変更を検知できない（R-02 の緩和策が機能しなくなる）
- テストデータはコア層の Factory / Fixture を共用する

### デプロイ構成

UI app がコア層に組み込まれるため、**デプロイは単一の Django プロジェクトとして行う**。

| 項目 | 方針 |
|------|------|
| **デプロイ単位** | コア層 + UI app を含む 1 つの Django プロジェクト |
| **静的ファイル** | `collectstatic` で収集し、Nginx 等で配信 |

## 5. Cluster 分割方針

### 用語定義

| 用語 | 定義 |
|------|------|
| **Cluster** | 関連する機能をまとめた実装単位。1 つの Cluster は 1 つ以上の Slice で構成される。設計書・Issue・Closure Audit の単位 |
| **Slice** | Cluster 内の最小実装単位。1 つの Slice が orchestrator → Cursor への 1 回の dispatch に対応する。precondition（開始条件）と postcondition（完了条件）を持つ |
| **Closure Audit** | Cluster 内の全 Slice が完了した後に行う結合面の検証。Slice 間の画面遷移・状態の整合性を確認する |

### 分割の基本方針

UI の Cluster は **UI 系統（スタッフ / オーナー）× 機能領域** で分割する。

| 方針 | 理由 |
|------|------|
| **スタッフ UI とオーナー UI は別 Cluster** | ロール・デバイス・設計哲学が異なる。同じ Cluster に混ぜると Cursor への指示が混乱する |
| **同じ機能でも UI 系統が違えば別 Cluster** | 例: 顧客管理はスタッフ UI（簡易表示）とオーナー UI（フル CRUD）で別テンプレート・別フォーム・別 UX。別 Cluster として独立に実装する |
| **API 側の Cluster と 1:1 にしない** | API は Model/Service 単位で分割されているが、UI は業務フロー・画面の凝集度で分割する。対応する API Cluster は依存関係として記載する |

### Slice 分割方針

各 Cluster は 1 本以上の Slice に分割する。Slice の粒度は基本設計で確定するが、方針は以下の通り。

| 方針 | 内容 |
|------|------|
| **画面フロー単位** | 1 Slice が「ユーザーの 1 つの操作フロー」を完結させる。例: 「顧客選択 → タスク表示 → 来店記録作成」は一連のフローなので同一 Slice |
| **独立してデモできる** | Slice 完了時点で、その範囲の画面がブラウザで動作確認できる |
| **テンプレート + View + Form がセット** | Slice には対応する View・Form・テンプレートを全て含める。「テンプレートだけ先に作る」はしない |

### 実行順序の考え方

| 優先度 | Cluster | 理由 |
|--------|---------|------|
| **1st** | US-01, UO-01（ログイン） | 認証がないと他の画面が動かない。API Phase 4 完了後すぐ着手可能 |
| **2nd** | US-02（接客フロー） | CRM のコアバリュー。スタッフが日常的に使う画面。これがないとリリースできない。API の着手条件は US-03 より遅い（Phase 7）が、Phase 7 完了後に最優先で着手する |
| **3rd** | US-03（顧客・来店簡易管理） | 接客フローの補助。API の着手条件は Phase 6 と早いが、US-02 が CRM の核であるため US-02 を優先する。US-02 完了後に着手 |
| **4th** | UO-02, UO-03（顧客・来店管理） | オーナーの管理業務。接客フローとは独立して進行可能 |
| **5th** | US-04, UO-04（マッチング） | Airレジ連携。API Phase 8 完了待ち |
| **6th** | UO-05（分析） | 来店データが溜まってから意味が出る。最後でよい |

### Closure Audit 方針

| 対象 | 検証内容 |
|------|---------|
| **Cluster 内** | Slice 間の画面遷移が途切れないこと。共通テンプレート（ナビゲーション等）が全画面で機能すること |
| **スタッフ UI 全体** | US-01 → US-02 → US-03 → US-04 の業務フローが一気通貫で動作すること |
| **オーナー UI 全体** | UO-01 → UO-02〜05 の全画面へのナビゲーションが機能すること |
| **UI ↔ API 結合面** | Service 層のエラー（バリデーション、権限、404 等）が UI に適切に伝搬・表示されること |

## 6. リスクと未決事項

### リスク

| # | リスク | 影響 | 緩和策 |
|---|--------|------|--------|
| R-01 | **Airレジマッチングの運用が定着しない** | 会計後にスタッフがマッチングを行わず、浮遊売上が常態化する。顧客別売上貢献度の分析精度が低下する | スタッフ UI でのマッチング操作を極力簡単にする（タップ 1〜2 回で確定）。浮遊売上は「日別店舗売上」としては残るため、売上総額の把握には支障がないことをオーナーに説明する。未マッチ件数をオーナー UI に表示し、後から精査できる導線を用意する |
| R-02 | **コア層 Service 層のインターフェース変更で UI が壊れる** | UI はコア層の Service 層を直接呼び出す構成のため、Service のシグネチャや戻り値が変わると UI View が壊れる | Service 層の変更は API Cluster の設計レビューで検知する。UI テスト（Django TestClient）で Service 呼び出しの正常系を常時検証する。コア層の変更 PR に UI 影響チェックを含める運用を導入する |
| R-03 | **ヘッドレス原則との緊張関係がレビューで論点になる** | UI が Service 層を直接呼ぶ構成は「API 経由」の原則と異なる。レビューアーが問題視する可能性がある | セクション 1 で判断理由を明文化済み。「設計原則は守り通信経路は合理的に選択した」という立場。API エンドポイントは引き続き公開され、他の消費者（CLI, LLM, LINE Bot）が利用可能であることを説明する |
| R-04 | **タブレット共有時の操作者取り違え** | 前のスタッフがログアウトせずに次のスタッフが操作し、来店記録の `staff` が間違ったスタッフに紐づく | MVP の緩和策: (1) ナビゲーションに現在の操作者名を常に表示し、取り違えに気づけるようにする (2) 一定時間操作がない場合の自動ログアウト（セッションタイムアウト）を設定する (3) 切替は「ログアウト → QR 再ログイン」で統一し、手順を明確にする（D-02 参照）。簡易切替 UI による心理的ハードル低減は Phase 2 で検討 |

### 未決事項
